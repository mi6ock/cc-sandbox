# Goal/Plan Manager

タスクのゴールドリフトを防ぐためにGoalファイルとPlanファイルをブランチ単位で管理する。

## 背景: なぜこのスキルが必要か

LLMエージェントは長いコンテキストの中で初期のゴールを忘れやすい（lost-in-the-middle問題）。
さらにcontext compaction後は元のゴール指示が要約に埋もれて失われる。
これにより「手順は覚えてるが何のためにやっていたか忘れた」状態が発生し、タスクを途中で完了扱いにしてしまう。

このスキルはGoal（何を達成するか）とPlan（どうやるか）を外部ファイルに永続化し、
hooksで自動再注入することでこの問題を解決する。

## コンセプト

- **Goalファイル**: ユーザーの元の要求・完了条件・スコープ外を記録。作成後は変更しない
- **Planファイル**: フェーズ・手順・チェックボックスを記録。進捗に応じて更新する
- **ブランチ名で紐付け**: セッション跨ぎでも同じブランチなら同じファイルを参照
- **Goalファイル作成がopt-in**: Goalを作らなければhook群は全部透過。短いタスクには影響しない
- **Goalが起点**: Goal → Plan → 実装 の順序を守る。Goalなしにplanを作らない

## ファイル配置

```
.claude/
├── goals/{ブランチ名}.md   # ゴール定義（変更禁止）
├── plans/{ブランチ名}.md   # プラン定義（進捗更新あり）
├── goals/_example.md       # テンプレート
├── plans/_example.md       # テンプレート
├── hooks/
│   ├── inject-context.sh   # GOAL+PLAN再注入
│   ├── check-plan-goal.sh  # ファイル存在・順序チェック
│   └── verify-completion.sh # 未完了チェック+停止ブロック
└── settings.json            # hook設定
```

ブランチ名の `/` は `-` に置換する（例: `feature/add-retry` → `feature-add-retry`）。

## タスク開始フロー

この順序を厳守すること:

1. 現在のブランチを確認する。main/masterの場合はタスク用ブランチを切る。すでに作業ブランチにいる場合はそのまま進める
2. **最初にゴールファイルを作成する**（プランより先。必ず先）
3. ゴールに基づいてプランファイルを作成する
4. 実装を開始する

Goalファイルが存在しない状態でPlanファイルを作成してはならない。
Goalファイルが存在しない状態でコード変更を行ってはならない。

## Goalファイルの書き方

`.claude/goals/{ブランチ名}.md` に作成する。

含める内容:
- **ユーザーの要求**: ユーザーが伝えた元の指示を原文に近い形で記載する
- **完了条件**: 具体的・検証可能な形でチェックボックスとして列挙する
- **スコープ外**: 明示的にやらないことを列挙する

作成後は内容を変更しない（ユーザーから明示的な変更指示がない限り）。

テンプレート（`.claude/goals/_example.md` 参照）:

```markdown
## ユーザーの要求
（ユーザーが伝えた元の指示をここに記載）

## 完了条件
- [ ] 条件1
- [ ] 条件2
- [ ] 条件3

## スコープ外
- やらないこと1
- やらないこと2
```

## Planファイルの書き方

`.claude/plans/{ブランチ名}.md` に作成する。

含める内容:
- フェーズ分けされた手順
- 各ステップのチェックボックス
- 技術的なアプローチの概要

**TodoWriteは使用しない。このファイルで管理する。**
ステップ完了ごとにチェックボックスを `[x]` に更新する。

テンプレート（`.claude/plans/_example.md` 参照）:

```markdown
## Phase 1: （フェーズ名）
- [ ] ステップ1
- [ ] ステップ2

## Phase 2: （フェーズ名）
- [ ] ステップ3
- [ ] ステップ4
```

## タスク完了時

完了と判断する前に必ず:

1. Goalファイルの完了条件を1つずつ確認する
2. Planファイルの全ステップが `[x]` であることを確認する
3. 未達成の項目がある場合は完了とせず続行する

## Hooks

このスキルはhooksで自動的に動作する。セットアップは `scripts/setup.sh` を実行する。

| フック | タイミング | 動作 |
|--------|-----------|------|
| SessionStart (compact) | compaction後 | GOAL+PLAN再注入 |
| SessionStart (startup/resume) | セッション開始/再開 | GOAL+PLAN再注入 |
| Stop | タスク停止時 | 未完了チェック+停止ブロック |
| PreToolUse (AskUserQuestion) | 質問時 | 未完了チェック+質問ブロック |
| PostToolUse (Write/Edit) | コード変更時 | ファイル存在・順序チェック |

### Hookの挙動マトリクス

| Goal | Plan | 挙動 |
|------|------|------|
| なし | なし | 全hook透過（短いタスク/調査） |
| あり | なし | 正常（プラン作成中の段階） |
| あり | あり | 完了時チェック有効 |
| なし | あり | 順序違反として警告 |

## セットアップ

プロジェクトルートで以下を実行:

```bash
bash /path/to/goal-plan-manager/scripts/setup.sh
```

これにより:
- `.claude/goals/`, `.claude/plans/`, `.claude/hooks/` ディレクトリが作成される
- hookスクリプト3本がコピーされる
- テンプレートファイルがコピーされる
- `.claude/settings.json` にhook設定がマージされる
- 実行権限が付与される

### 手動セットアップ

setup.shを使わない場合は `scripts/` 内の各ファイルを手動で配置する。
詳細は各スクリプトファイル冒頭のコメントを参照。

## カスタマイズ

- **停止ブロックを無効化**: `verify-completion.sh` の `exit 2` を `exit 0` に変更すると警告のみ
- **.gitignore**: goals/plans をチームで共有しない場合は `.claude/goals/` `.claude/plans/` を追加
